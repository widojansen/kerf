### What's Missing to Complete the Architecture

1. **Telegram channel** — The only planned component not built. The architecture has a full sketch with polling, `fetch_updates`, and authorized sender filtering. WhatsApp proves the pattern works.

2. **Dynamic tool loading** — The architecture envisioned `Tools.DynamicSupervisor` for "custom skills loaded at runtime." The ETS-backed Registry already supports `register_tool/2` at runtime, so the foundation exists — but there's no plugin discovery/loading mechanism.

3. **Multi-provider LLM** — The architecture mentioned OpenRouter alongside Anthropic. Currently Anthropic-only.

### Summary

The project has **delivered more than the architecture planned** (telemetry, dynamic tool registry, web tools, SSRF protection) while making **pragmatic simplifications** (no unnecessary GenServers for stateless tools, no Config Agent, no Mox). The one gap is the **Telegram channel**, which was the originally planned "first real channel" but got deprioritized in favor of WhatsApp. The architecture's core thesis — OTP supervision, per-group isolation, security-first — is fully realized.

## ExClaw vs Elixir Agent Frameworks (Sagents & Legion)

### Philosophical Positioning

| | **ExClaw** | **Sagents** | **Legion** |
|---|---|---|---|
| **What it is** | Personal AI assistant (application) | Agent framework (library) | Agent framework (library) |
| **Agent paradigm** | Traditional tool calling | Traditional tool calling | Code generation in sandbox |
| **LLM integration** | Direct Req + Anthropic API | LangChain (multi-provider) | ReqLLM (multi-provider) |
| **Primary goal** | Always-on assistant with channels, memory, scheduling | Building interactive AI chat apps | Autonomous task execution |

ExClaw is an **end-user product**; Sagents and Legion are **libraries for building products**. ExClaw could theoretically be built on top of either framework, but it predates both and rolls its own agent loop.

### Feature-by-Feature Comparison

| Capability | **ExClaw** | **Sagents** | **Legion** |
|---|---|---|---|
| **Agent loop** | Custom GenServer per group | Middleware-wrapped LLMChain | Code-gen + sandbox eval |
| **Tool system** | ETS Registry + Dispatcher (5 built-in) | LangChain.Function structs | Any module with `use Legion.Tool` |
| **Multi-provider LLM** | Anthropic only | Anthropic, OpenAI, Google | OpenAI, Anthropic (via ReqLLM) |
| **Streaming** | No | Yes (delta broadcasting) | No |
| **Persistence** | SQLite (facts, messages, MEMORY.md) | Ecto generators (JSONB) | None |
| **Channel adapters** | CLI, WhatsApp (Telegram planned) | None (it's a library) | None (it's a library) |
| **Scheduling** | Cron/interval/once/at via OTP timers | None | None |
| **Container sandboxing** | Docker per group (hardened) | None | In-process AST sandbox |
| **Web dashboard** | Phoenix LiveView (localhost) | LiveView generators + PubSub | None |
| **Telemetry** | ClickHouse + JSONL fallback | PubSub events | `:telemetry` events |
| **Security layers** | FileGuard, ShellSandbox, PromptGuard, SSRF | HITL middleware | AST allowlist |
| **Human-in-the-loop** | None (user is always the initiator) | Deep (per-tool approve/reject/edit) | Basic (HumanTool.ask) |
| **Middleware system** | None | Rich sandwich pattern | None |
| **Sub-agents** | None | SubAgent middleware | AgentTool.call |
| **Structured output** | No (free-form text) | No | JSON schema enforced |
| **State management** | GenServer state + SQLite + filesystem | State struct with serialization | In-memory message context |
| **Supervision tree** | Full custom (12+ supervisors) | DynamicSupervisor + Registry | Simple GenServer |
| **Web tools** | WebFetch (SSRF-protected), WebSearch (SearXNG) | None built-in | None built-in |

### Where ExClaw is Stronger

1. **Infrastructure completeness** — ExClaw is a running system with channels, scheduling, containers, telemetry, and a dashboard. Neither framework provides any of this.

2. **Security depth** — Three dedicated security modules (file, shell, prompt) plus SSRF protection, Docker hardening (`--read-only`, `--cap-drop ALL`, `--network none`, `--pids-limit`), and path traversal prevention. Sagents relies on HITL approval. Legion relies on AST analysis.

3. **Persistent memory** — Three storage types (facts, MEMORY.md, message history) in SQLite. Sagents has Ecto generators but no memory abstraction. Legion has nothing.

4. **Container isolation** — Real Docker sandboxing per group. Sagents has a virtual filesystem (in-process). Legion evaluates code in a BEAM sandbox (same process, AST-restricted).

5. **Channel adapters** — WhatsApp via Baileys, CLI, Telegram planned. Neither framework handles messaging channels at all.

### Where ExClaw is Weaker

1. **No middleware/plugin system** — Sagents' sandwich middleware pattern is genuinely powerful for composability. ExClaw's tool_executor injection is functional but not a general extensibility mechanism.

2. **No HITL** — Sagents has granular per-tool human approval with interrupt/resume. ExClaw has no approval workflow — the user initiates, the agent executes.

3. **No streaming** — Sagents broadcasts LLM deltas in real-time. ExClaw is synchronous request/response throughout.

4. **Single LLM provider** — Both frameworks support multiple providers. ExClaw is locked to Anthropic.

5. **No sub-agents** — Both frameworks support multi-agent orchestration. ExClaw has one agent per group, no delegation.

6. **No structured output** — Legion enforces JSON schemas on agent responses. ExClaw returns free-form text.

### What ExClaw Could Adopt

| From | Feature | Effort | Value |
|------|---------|--------|-------|
| **Sagents** | Middleware system | High | Would enable HITL, conversation summarization, auto-titles |
| **Sagents** | LLM streaming | Medium | Better UX for dashboard and WhatsApp |
| **Sagents** | HITL interrupts | Medium | Critical for safety with autonomous tools |
| **Legion** | Structured output schemas | Low | Better tool result parsing, typed responses |
| **Legion** | Code-gen sandbox as a tool | Medium | Powerful "eval Elixir" tool alongside existing tools |
| **Both** | Multi-provider LLM | Medium | OpenRouter/OpenAI as fallback providers |

### Bottom Line

ExClaw occupies a different niche. It's a **complete, deployed personal assistant** with infrastructure neither framework provides (channels, scheduling, containers, telemetry, persistent memory). The frameworks are more sophisticated in their agent loop design (Sagents' middleware, Legion's code-gen), but they're building blocks, not products.

The most valuable things to bring into ExClaw would be **streaming** (Sagents-style, for real-time UX) and **HITL approval** (for safety when tools have side effects). Legion's code-generation approach is interesting but orthogonal — it could be added as an optional tool rather than replacing the existing dispatch model.

Key decisions needed for the split that the plan doesn't fully address:

1. **Telemetry** — Should `ExClaw.Telemetry.Logger` be in the library or the app? The plan lists it as library ("Future: ClickHouse event logging"), but it has ClickHouse/JSONL-specific infrastructure. Host apps may want their own telemetry backends.

2. **Container sandboxing** — Not mentioned in the plan's library/app split at all (it was built after the plan). Docker containers are app-specific infrastructure, so they belong in `exclaw_app`, but the tool dispatch pattern needs the container manager injected.

3. **Tool Registry** — The plan doesn't account for the ETS-backed dynamic registry (Phase 11). The Registry is a GenServer that both the library agent loop and app-specific tools need. It could go either way — library (so host apps get tool registration for free) or app (each host manages its own registry).

4. **WhatsApp** — Not in the plan's split table. Clearly belongs in `exclaw_app`, but its supervisor pattern could inform future channel adapters in host apps.

5. **Test count mismatch** — The plan expects ~162 tests total. Current reality is **367 tests + 1 doctest**. The split needs re-estimation.

## Strategic Decision: How to Continue

### The Core Tension

Three goals are pulling in different directions:

1. **Ship AI to izi_monitoring and tecdoc_importer** — needs a library, soon
2. **ExClaw as a standalone assistant** — needs channels, scheduler, containers
3. **Better agent capabilities** — middleware, HITL, streaming, sub-agents, code-gen

The umbrella restructuring (Integration Plan Phase 1) serves goal 1 but doesn't advance goals 2 or 3. Adding Sagents/Legion features serves goal 3 but delays goals 1 and 2.

### Context: Build Velocity

ExClaw was built in **~24 hours of active work** over 2 days (Feb 18-20, 2026):

| Day | Phases | Tests Added |
|-----|--------|-------------|
| Feb 18 | Phase 1 (Security) | 61 |
| Feb 19 | Phases 2-7 (LLM, Agent, Memory, CLI, Scheduler, Dashboard) | 173 |
| Feb 20 | Phases 8-11 (Telemetry, Container, WhatsApp, Tool Registry) | 133 |

49 source files, ~5,200 lines of implementation, ~5,200 lines of tests, 367 tests passing with 0 warnings. This velocity was enabled by the architecture doc and CLAUDE.md defining all contracts upfront, with strict Red-Prompt-Green-Refactor discipline throughout.

### Three Paths Evaluated

**Path A: ExClaw stays the agent core — cherry-pick features in**

Keep the existing agent loop. Add middleware, streaming, and code-gen incrementally. Then do the umbrella split.

- **Pro:** Full control, no external dependencies, 367 tests stay valid, security model unchanged.
- **Con:** Rebuilding what Sagents already ships. Middleware, HITL, sub-agents, streaming — that's significant work. Delays the integrations further.

**Path B: Replace ExClaw's agent core with Sagents**

Sagents is designed to be exactly the library the Integration Plan tries to extract. It already has middleware, HITL, sub-agents, streaming, LiveView integration, and persistence — with no `mod:` key, meant to be added to Phoenix apps.

ExClaw's unique value becomes a set of Sagents middleware and tools:

| Sagents (agent core) | ExClaw (infrastructure layer) |
|----------------------|-------------------------------|
| Agent loop | Security middleware (FileGuard, ShellSandbox, PromptGuard) |
| Middleware system | Container tools (Docker sandboxing) |
| HITL interrupts | Memory middleware (facts, MEMORY.md, messages) |
| Sub-agents | Telemetry middleware (ClickHouse/JSONL) |
| Streaming | Channel adapters (CLI, WhatsApp, Telegram) |
| LiveView integration | Scheduler |
| Persistence (Ecto) | Web tools (WebFetch, WebSearch with SSRF protection) |
| | Tool Registry (ETS-backed) |

- **Pro:** Gets middleware, HITL, streaming, sub-agents, LiveView integration immediately. Focus on domain value (security, containers, memory) not plumbing.
- **Con:** External dependency. Sagents uses LangChain (another dep). Existing 367 tests need significant rework. Security-check-before-tool-execution flow needs to map onto Sagents' middleware model.

**Path C: Hybrid — Sagents for the integrations, ExClaw stays standalone**

Use them side by side. Don't merge.

- **izi_monitoring** gets Sagents directly (LiveView chat + HITL + streaming — Sagents' sweet spot)
- **tecdoc_importer** gets Sagents directly (same reasoning)
- **ExClaw standalone** stays as-is for the personal assistant use case (CLI, WhatsApp, containers, scheduler)
- Share security modules and tool patterns as a thin utility library

- **Pro:** Fastest path to shipping AI in the two target projects. No restructuring needed. ExClaw doesn't regress. Learn Sagents in a real context before committing.
- **Con:** Two agent systems to maintain. Security patterns duplicated or extracted into a shared lib. ExClaw doesn't get middleware/HITL/streaming unless built separately.

### Assessment

**Path C is the pragmatic move.**

- The two target projects need a **LiveView-integrated agent library with streaming**. That's exactly what Sagents is. Extracting ExClaw as a library would be rebuilding Sagents.
- ExClaw as a standalone assistant has different needs (channels, containers, scheduler, always-on) that Sagents doesn't solve and isn't designed for.
- A **1-2 day spike with Sagents in izi_monitoring** will validate whether Path B (full adoption) makes sense long-term, without risking ExClaw.
- Legion's code-generation sandbox can be added to either system as a tool — it's orthogonal to this decision.
- The integration plan's umbrella restructuring was the right idea when ExClaw was the only Elixir agent option. Now that Sagents exists and is purpose-built for the "add AI to Phoenix apps" use case, the calculus changes.

### Recommended Next Steps

1. **Spike Sagents in izi_monitoring** (1-2 days) — wire up one ClickHouse query tool, get NL→SQL→explanation working in a LiveView chat component.
2. **Evaluate** — does Sagents' middleware model accommodate ExClaw's security checks? Does streaming work well? Is LangChain acceptable as a dependency?
3. **If yes:** Port ExClaw's security modules as Sagents middleware for both target projects. ExClaw standalone continues evolving independently.
4. **If no:** Fall back to Path A — cherry-pick the patterns into ExClaw and proceed with the umbrella restructuring.

Either way, ExClaw's 11 phases of work aren't wasted. The security layer, memory system, telemetry, container sandboxing, and channel adapters are infrastructure that no framework provides. The question is only whether ExClaw's agent loop or Sagents' agent loop is the right foundation for the library use case.
