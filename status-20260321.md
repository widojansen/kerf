## What Was Built

### 1. vLLM Backend (replacing Ollama for production inference)

**Why:** Multiple consumers (OpenClaw/Anita, gmail-triage, health_monitor, incoming_mail, ExClaw) hitting the Spark simultaneously. Ollama processes requests sequentially — at 10+ concurrent users, response times degrade from 2s to 45s+. vLLM's continuous batching and PagedAttention maintain throughput under concurrent load (16x throughput advantage on Blackwell GPUs).

**What was built:**

- `lib/exclaw/llm/vllm_provider.ex` — GenServer speaking OpenAI `/v1/chat/completions` format
  - Tool calling with Anthropic → OpenAI format conversion (transparent to callers)
  - Empty `tool_calls: []` handled as text response (vLLM quirk)
  - 120s `receive_timeout` for thinking models
  - Bearer token auth header (vLLM accepts any value)
  - 8 unit tests covering text, format, system prompts, tools, tool call parsing, errors, rate limiting
- `lib/exclaw/llm/supervisor.ex` — Updated to support three backends:
  - `LLM_BACKEND=vllm` → starts VLLMProvider (production, concurrent)
  - `LLM_BACKEND=ollama` → starts OllamaProvider (legacy, single-user)
  - unset → Anthropic-only mode
  - vLLM preferred over Ollama when both configured
- `lib/exclaw/llm/model_router.ex` — Updated dispatch to handle 3 providers
  - Module detection via `$initial_call` process dictionary
  - Routes `nvidia/*`, `qwen*`, `deepseek*`, `glm*`, `nemotron*`, `llama*`, `phi*` to local backend
  - Routes `claude-*` to Anthropic, fallback to Anthropic
- `config/runtime.exs` — New env vars:
  - `LLM_BACKEND=vllm` activates VLLMProvider
  - `VLLM_URL` (default `http://localhost:8000`)
  - `VLLM_MODEL` (default `nvidia/Qwen3-32B-NVFP4`)

**Spark deployment:**

- vLLM Docker container: `nvcr.io/nvidia/vllm:26.02-py3` (14.7GB image)
- Model: `nvidia/Qwen3-32B-NVFP4` (NVFP4 quantization, native Blackwell FP4 tensor cores)
- Flags: `--gpu-memory-utilization 0.7 --enable-auto-tool-choice --tool-call-parser hermes`
- KV cache: 496K tokens, 12x concurrency at 40K context
- Systemd service installed: `sudo systemctl enable vllm`
- End-to-end verified: ExClaw CLI → VLLMProvider → vLLM → Qwen3-32B-NVFP4 → "4"

### 2. Telegram Channel Adapter (Phase 6)

**What was built:**

- `lib/exclaw/channels/telegram.ex` — GenServer with long polling
  - Polls Telegram Bot API `getUpdates` with configurable interval
  - Authorization via `allow_from` user ID list (empty = allow all)
  - Derives group_id as `tg_{chat_id}` for per-chat Agent.Session isolation
  - Strips `<think>...</think>` tags from vLLM/Qwen responses before sending
  - Message truncation at 4096 chars (Telegram limit)
  - Exponential backoff on poll errors (up to 30s)
  - Async message processing via Task.Supervisor (polling continues while agent thinks)
  - Persists exchanges to Memory.Store
  - 17 unit tests covering all pure functions
- `lib/exclaw/channels/telegram/supervisor.ex` — one_for_one supervisor
- `lib/exclaw/application.ex` — Updated:
  - Added `{Task.Supervisor, name: ExClaw.TaskSupervisor}` for async Telegram message handling
  - Added `telegram_children()` conditional startup (same pattern as WhatsApp)
- `config/runtime.exs` — New env vars:
  - `TELEGRAM_BOT_TOKEN` — token from @BotFather
  - `TELEGRAM_ALLOW_FROM` — comma-separated Telegram user IDs

**To start ExClaw with Telegram on the Spark:**

```bash
export LLM_BACKEND=vllm
export EXCLAW_DEFAULT_MODEL=nvidia/Qwen3-32B-NVFP4
export TELEGRAM_BOT_TOKEN=your_bot_token_from_botfather
export TELEGRAM_ALLOW_FROM=your_telegram_user_id
mix run --no-halt
```
